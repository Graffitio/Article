죽지 않았다

GPT-5.5와 Claude Opus 4.7 시대,
무엇이 바뀌었고 지금 당장 뭘 바꿔야 하는가

Guide · Prompt Strategy · 2026

프롬프트 쓰는 법이 달라졌다는 말이 돌고 있다. 맞는 말이다. 하지만 프롬프트가 없어진 게 아니다. 사라진 건 낡은 방식이다. 지금도 많은 사람이 옛날 방식 그대로 AI를 쓰고 있다. 어떤 게 바뀌었는지 정확히 알면 바로 써먹을 수 있다.

먼저, 무엇이 약해졌나

최신 모델이 등장하면서 예전에 잘 먹혔던 프롬프트 패턴들이 힘을 잃었다. 대표적인 것들이 있다.

예전 방식 — 지금은 노이즈

"전문가처럼 답해줘."

"천천히, 단계별로 생각해."

"무조건 검색해. 반드시 도구를 써."

"이렇게 → 그다음 → 마지막으로 이렇게 해라."

지금 방식 — 실제로 작동하는 것

원하는 결과물의 형태를 명확히 정의

성공 기준과 종료 조건을 제시

언제 도구를 써야 하는지 상황을 알려줌

맥락·예시·업무 규칙을 구조적으로 제공

모델이 똑똑해졌기 때문이다. 이제 모델은 답변만 하지 않는다. 파일을 읽고, 검색하고, 코드를 고치고, 여러 단계를 이어서 실행한다. 답변하는 모델에는 좋은 질문이 필요했지만, 일하는 모델에는 작업 규칙이 필요하다.

모델을 설득하는 시대가 끝났다. 모델이 일할 기준을 세우는 시대가 됐다.

GPT-5.5 — 결과 계약서를 써라

OpenAI가 GPT-5.5 프롬프트 가이드에서 강조하는 핵심은 하나다. "어떻게 해라"가 아니라 "어디까지 가면 끝인가"를 써라.

GPT-5.5는 짧고 목적 중심인 프롬프트에서 더 잘 작동한다. 예전 모델을 위해 쌓아둔 긴 단계별 프롬프트를 그대로 가져오면 오히려 모델이 스스로 풀 수 있는 길을 좁히고, 답변을 기계적으로 만든다.

약한 프롬프트

"고객 문의 처리해줘."

강한 프롬프트

"환불 가능 여부를 판단하고, 정책 근거를 붙이고, 사용자가 바로 이해할 수 있는 답변으로 마무리해줘. 근거가 부족하면 질문은 하나만 해."

Stop Rules — 언제 멈출지도 프롬프트다

GPT-5.5 가이드에서 중요하게 다루는 개념이다. 최신 모델은 도구를 쓰고, 파일을 읽고, 코드를 고치는 과정에서 너무 오래 일하는 문제가 생긴다. 좋은 프롬프트는 AI를 더 오래 일하게 하는 문서가 아니다. 제때 멈추게 하는 문서다. "근거가 부족하면 질문은 하나만 해" 같은 문장이 Stop Rule이다.

또한 GPT-5.5에서는 프롬프트 문장만이 품질을 결정하지 않는다. reasoning effort, verbosity, structured output, tool description 같은 설정 전체가 모델의 작업 환경을 만든다.

Claude Opus 4.7 — 업무 매뉴얼을 써라

Anthropic 쪽의 메시지는 다른 결이다. 핵심 비유는 이것이다. "똑똑한 신입 직원에게 회사 규칙을 알려줘라."

Claude는 똑똑하다. 하지만 당신 팀의 보고서 형식은 모른다. 말투도 모른다. 어디까지 알아서 해야 하는지도 모른다. "알아서 잘해줘"가 아니라 "우리 팀에서는 이렇게 일해"를 알려줘야 한다.

Effort 레벨 — 프롬프트보다 먼저 확인할 것

Anthropic은 코딩·에이전트 작업에는 xhigh effort부터, 지능이 필요한 작업은 최소 high를 권장한다. 복잡한 일을 맡겼는데 답이 얕다면 프롬프트를 더 꾸미기보다 effort를 올리는 게 먼저일 수 있다. 단, 더 깊게 생각하게 하면 느려지고 토큰을 더 쓴다. "얼마나 생각하게 할지"도 중요한 설계 결정이다.

Opus 4.7은 이전 버전보다 도구를 덜 쓰고 추론을 더 한다. 예전에 "무조건 검색해"처럼 도구 사용을 강제하는 문장을 많이 넣었다면, 이제 그 문장이 불필요한 도구 호출을 늘릴 수 있다. "이런 상황에서는 도구를 써"처럼 조건을 구체적으로 써야 한다.

Opus 4.7은 말을 더 문자 그대로 따른다. "이 규칙을 첫 번째 섹션에 적용해"라고 쓰면 다른 섹션에는 적용하지 않을 수 있다. 모든 섹션에 적용하려면 "모든 섹션에 적용해"라고 명시해야 한다. 모호한 프롬프트가 바로 티가 난다.

XML 태그 구조도 계속 유효하다. <instructions><context><input><examples> 형태로 나눠주면 Claude가 지시인지, 배경인지, 예시인지를 덜 헷갈려한다. 또 긴 문서를 줄 때는 문서를 위에, 질문을 아래에 배치하는 것이 응답 품질을 높인다.

Opus 4.7

둘의 공통된 방향

겉으로는 달라 보이지만, 두 가이드가 같이 말하는 것이 있다.

  • 낡은 패턴을 걷어내라. OpenAI는 과정이 과하게 적힌 오래된 프롬프트를 줄이라 하고, Anthropic은 과한 도구 강제 문구를 줄이라고 한다.
  • 모델이 강해질수록 나쁜 프롬프트가 더 위험하다. 옛날 모델은 이상한 지시를 대충 무시했다. 지금 모델은 정밀하게 따른다. 좋은 지시도 잘 따르지만, 나쁜 지시도 잘 따른다.
  • 프롬프트의 형태가 바뀌었다. AGENTS.md, CLAUDE.md, WORKFLOW.md처럼 파일이 됐다. 대화창에 매번 입력하는 게 아니라 작업 규칙을 문서로 만드는 방향이다.
  • AI에게 일을 맡기려면, 일의 기준부터 써야 한다. 둘 다 같은 말을 하고 있다.

프롬프트는 사라진 게 아니다. 이름이 바뀌었다. 파일이 됐고, 규칙이 됐고, 운영 문서가 됐다.

지금 당장 바꿀 것

기존 프롬프트를 쓰고 있다면, 아래 순서로 점검해보면 된다.

  • GPT-5.5를 쓴다면 — 과정을 서술한 긴 프롬프트를 줄이고, 원하는 결과물의 형태와 종료 기준을 명확히 써라. "이렇게 해"보다 "이 기준을 만족하면 끝"이다.
  • Opus 4.7을 쓴다면 — "무조건 검색해"처럼 도구를 강제하는 문장을 점검해라. Effort 레벨을 먼저 확인하고, 맥락과 예시를 XML 태그로 구분해서 줘라.
  • 둘 다 해당 — 모호한 문장을 없애라. 지금 모델들은 문자 그대로 따른다. 범위를 명확히 쓰지 않으면 기대와 다른 결과가 나온다.
  • 반복 사용하는 프롬프트라면 — 대화창에 붙여넣는 대신 파일로 만들어서 관리하는 방향을 고려해라. AGENTS.md, CLAUDE.md 같은 구조가 그 방향이다.
